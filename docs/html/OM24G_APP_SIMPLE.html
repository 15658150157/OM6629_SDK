om24g_app_simple

Building

The Device example is under SDK\projects\om24g_app_simple in the SDK folder structure. Perform the following steps to set up an application to send packets:

1、Initialize 2.4GHz Proprietary protocol using drv_om24g_init(const om24g_config_t *cfg). You can use the default parameters in om24g_config as starting point for the cfg parameter and reconfigure them if needed.

2、If necessary, use any of the folowing functions to update the addresses, the channel, and the bitrate:

  • drv_om24g_set_addr(uint8_t pipenum, uint8_t addr)
  • drv_om24g_set_freq(uint16_t freq, float fract_freq)
  • drv_om24g_set_rf_parameters(om24g_rate_t data_rate, uint16_t frequency, float fract_freq)
Note
drv_om24g_set_rf_parameters must be used to configure the rate, which has configured the frequency offset, but drv_om24g_set_rate has not configured the frequency offset.

3、drv_om24g_control(OM24G_CONTROL_CLK_ENABLE, NULL) can turn on the 24G clock，drv_om24g_control(OM24G_CONTROL_CLK_DISABLE, NULL) can turn off the 24G clock. The clock has been turned on by om24g_init by default.

4、Start sending packets:

  • Registering Callbacks by calling drv_om24g_register_event_callback(om24g_callback).
  • Add packets to the TX FIFO by calling drv_om24g_write_int(uint8_t *tx_payload, uint8_t tx_num).
  • Wait for an event interrupt in the callback function.
    • in no ack mode, wait until event DRV_EVENT_COMMON_TRANSMIT_COMPLETED represents successful transmission.
    • In ack mode, event DRV_EVENT_COMMON_TRANSMIT_COMPLETED is triggered only when a response packet is received, indicating successful transmission. Otherwise, the transmission fails and triggers event DRV_EVENT_OM24G_MAX_RT(max retry).

5、Start receiving packets:

  • Registering Callbacks by calling om24g_register_event_callback(om24g_callback).
  • Call drv_om24g_read_int(uint8_t *rx_payload) to turn on the RF and start receiving.
    • in no ack mode，handle DRV_EVENT_COMMON_RECEIVE_COMPLETED events as packets are coming in.
    • In ack mode, to attach payloads to acknowledgment packets, add them to the TX FIFO using drv_om24g_write_ack(uint8_t tx_num). The payload must be queued after a packet is received. After a queued payload is sent with an acknowledgment, only after receiving the next new packet with different PID values, a DRV_EVENT_COMMON_TRANSMIT_COMPLETED event is only queued.

Example

TX Device

This example demonstrates a basic 2.4G private protocol receiving packets. It enters continuous reception mode to listen for airborne packets. This demo has a speed of 1M and a frequency of 2478Mhz The package structure consists of a four byte preamble, a four byte synchronization word, a one byte header, a 0-64 byte payload (which can be configured with static_len to allow maximum length), and a two byte CRC.

The following are specific configurations for TX devices:

  • This chip can interconnect with 24g chips from multiple manufacturers and is compatible with om662x, nordic, TI, siliconlabs, default compatible with om662x, can be modified through the macro definition RTE_OM24G_RF_MODE in RTE_om6626.h (0: Compatible with om66XX 1: Compatible with TI2640 2: Compatible with SILICONLAB 3: Compatible with NORDIC).
  • Modify the macro definition TX_ROLE in app_om24g.c to 1 representing the sending device.
  • Call osThreadNew(om24g_thread, NULL, &Om24gThreadAttr) to create a 24G thread task. In the thread, call om24g_tx_test(OM24G_INT_STRUCTURE_B) to send a package of package format B in interrupt mode. In callback function om24g_callback(), waiting until event DRV_EVENT_COMMON_TRANSMIT_COMPLETED is triggered represents successful sending. Then call drv_om24g_control(OM24G_CONTROL_CLK_DISABLE, NULL) to turn off the RF clock to save power.
  • In the sending completion event, drv_pmu_timer_trig_set(PMU_TIMER_TRIG_VAL0, (drv_pmu_timer_cnt_get() + PMU_TIMER_MS2TICK(500))) can be called to trigger the next packet transmission at a timing of 500ms.
  • In the timer1_callback(), call drv_om24g_control(OM24G_CONTROL_CLK_ENABLE, NULL) to turn on the RF clock, and om24g_write_it() to send the next packet.

RX Device

This example demonstrates a basic 2.4G private protocol transmission packet. It sends a 32 byte payload in packet structure B every 500ms, for a total of 100 packets. The payload content ranges from 0 to 31. This demo has a speed of 1M and a frequency of 2478Mhz The package structure consists of a four byte preamble, a four byte synchronization word, a one byte header, a 0-64 byte payload (which can be configured with static_len to allow maximum length), and a two byte CRC.

The following are specific configurations for RX devices:

  • This chip can interconnect with 24g chips from multiple manufacturers and is compatible with om662x, nordic, TI, siliconlabs, default compatible with om662x, can be modified through the macro definition RTE_OM24G_RF_MODE in RTE_om6626.h (0: Compatible with om66XX 1: Compatible with TI2640 2: Compatible with SILICONLAB 3: Compatible with NORDIC).
  • Modify the macro definition TX_ROLE in app_om24g.c to 0 representing the receiving device.
  • Call osThreadNew(om24g_thread, NULL, &Om24gThreadAttr) to create a 24G thread task. In the thread, call om24g_rx_test(OM24G_INT_STRUCTURE_B) to receive package of format B in interrupt mode. In callback function om24g_callback(), waiting until event DRV_EVENT_COMMON_RECEIVE_COMPLETED is triggered represents successful receiving.

Running and Testing

After programming the TX Device example(TX_ROLE = 1) on one of the development kits and the RX device example(TX_ROLE = 0) on the other kit, test them by performing the following steps:

  • Connect the serial ports GPO5,6 to the upper computer of the serial assistant.
  • First power on the RX device, then power on the TX device.
  • Observe the logging output for both kits.
